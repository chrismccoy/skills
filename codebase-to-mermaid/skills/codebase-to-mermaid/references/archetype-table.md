# Archetype Table - Phase 2 Classification

Pick the project archetype based on the Phase 1 evidence. If two archetypes both fit, emit two diagrams.

| Archetype | Signals | Best diagram |
| --- | --- | --- |
| HTTP request handler | Routes + controllers + middleware | `sequenceDiagram` per route |
| CLI tool | Single entry → subcommands → side effects | `flowchart TD` |
| Data pipeline / ETL | Stage chain (ingest → transform → load) | `flowchart LR` |
| Event-driven / pub-sub | Producers, brokers, consumers | `flowchart LR` + `sequenceDiagram` |
| OO domain model | Many classes, inheritance, aggregates | `classDiagram` |
| Frontend SPA (React/Vue/Svelte) | `src/main.*` → router → pages → components → state stores → fetchers | `flowchart TD` (route tree) + `flowchart LR` (data fetch path) |
| Next.js / Nuxt / Remix | `app/`, `pages/`, `routes/` dirs; server components / loaders / actions | `flowchart TD` route tree split by server vs client + `sequenceDiagram` for a sample request |
| Node backend (Express/Nest/Fastify/Koa/Hono) | App bootstrap → middleware chain → routers → controllers → services → repos/ORM → DB driver | (a) `flowchart LR`: `index.js`/`server.js` → middleware → router → handler → service → ORM/query → DB (label DB nodes with the actual engine: `Mongo`, `MySQL`, `Postgres`, `SQLite`, `Redis`). (b) `sequenceDiagram` for one representative route covering request validation, ORM call, response. (c) `classDiagram` of ORM models if `prisma`/`drizzle`/`typeorm`/`mongoose` schemas drive the data layer. (d) If a queue (`bullmq`, `agenda`, `kafkajs`) is present, add `sequenceDiagram` of producer → broker → worker → DB |
| Monorepo / multi-service | Multiple package roots | `flowchart LR` (service map) + per-service drill-down |
| Generic plugin / hook system | Event registry + listener registration | `sequenceDiagram` (hook lifecycle) |
| **WordPress plugin** | `Plugin Name:` header, `add_action`/`add_filter`, REST routes, admin pages, CPTs, shortcodes | (a) `flowchart TD` lifecycle: activation → `plugins_loaded` → `init` → `admin_init`/`rest_api_init` → frontend output. (b) `sequenceDiagram` for each AJAX/REST endpoint and admin form POST. (c) `flowchart LR` of registered hooks → callbacks → side effects (DB, mail, HTTP) |
| **WordPress theme** | `style.css` header, `functions.php`, template hierarchy, enqueues, `theme.json`, optional FSE `templates/` | (a) `flowchart TD` of template hierarchy resolution per request type (single/archive/page/front-page). (b) `flowchart LR` of `functions.php` hook registrations → callbacks. (c) `flowchart TD` of enqueued assets per template |
| **WP block plugin (Gutenberg)** | `block.json`, `registerBlockType`, JS `src/blocks/*`, PHP `render_callback` | `flowchart LR`: build (`@wordpress/scripts`) → registered block → editor (`edit`/`save`) ↔ server render (`render_callback`) |
| **WooCommerce extension** | Hooks prefixed `woocommerce_`, custom payment/shipping classes, cart/checkout filters | `sequenceDiagram` of checkout/cart flow with extension's hook insertions highlighted |
| **Laravel app** | `artisan`, `routes/*.php`, `app/Http/Controllers/`, providers, Eloquent models | (a) `flowchart LR`: request → kernel → middleware stack → route → controller → service → Eloquent model → DB. (b) `flowchart TD` of service providers boot order + bound bindings. (c) `sequenceDiagram` for queued jobs / events / listeners if `app/Events/` non-empty |
| **Symfony app** | `bin/console`, `config/routes.yaml`, controllers w/ `#[Route(`, DI container, EventDispatcher | (a) `flowchart LR`: HttpKernel → event listeners (`kernel.request` → `kernel.controller` → `kernel.response`) → controller → service → Doctrine repo → DB. (b) `flowchart TD` of bundle registration + service container highlights. (c) `sequenceDiagram` for Messenger handlers if `src/MessageHandler/` non-empty |
| **Other PHP MVC** | Slim/Lumen/Cake/Yii/CI/Drupal/Magento markers from Phase 1 | `flowchart LR` of request → router → controller → model → view layer using framework's actual class names |
| **Custom / vanilla PHP** | Front controller `index.php`, hand-rolled router or `switch($_GET['page'])`, raw `PDO`/`mysqli`, `include`/`require` chains | (a) `flowchart TD`: `index.php` → routing → dispatched handler → included files → DB calls → rendered template. (b) `flowchart LR` of include graph (top 20 includes by inbound edges). Note explicitly that no framework was detected. |
| **Bash script / collection** | `*.sh` files with shebangs, `bin/`, `scripts/`, `Makefile`/`justfile`, `set -euo pipefail`, external tool invocations (curl/aws/docker/kubectl/git/mysql/etc) | (a) `flowchart TD` per script: entry → guards (root check, env check, `getopts`) → functions → branches (`if`/`case`) → external commands → exit codes. (b) `flowchart LR` of script-to-script call graph when scripts `source` or `exec` each other (or are wired by a `Makefile` target DAG). Edge labels = the actual invocation (`source ./lib.sh`, `bash deploy.sh prod`, `make build`). External side-effect nodes (e.g. `AWS S3`, `Docker Hub`, `MySQL`) shown as distinct shape (`stadium` or `cylinder`). |
| **WP-CLI script (Bash + WordPress)** | `.sh` files invoking `wp ` commands, optionally wrapping with `--path`, `--url`, `--ssh`, `--allow-root`; or `WP_CLI::add_command` in PHP | (a) `flowchart TD` per script: shebang → guards → ordered `wp` calls (each labeled with the full subcommand) → WordPress side effects (DB writes, file changes, cron events). (b) `sequenceDiagram`: Bash script → WP-CLI → WordPress bootstrap (`wp-load.php`) → plugin/theme hooks fired → DB / filesystem. (c) If custom `WP_CLI::add_command` exists, add `classDiagram` of command class + its registered subcommands. Annotate destructive steps (`wp db reset`, `wp search-replace`, `wp plugin delete`) with a distinct node style. |
| **Go HTTP service** | `go.mod`, `cmd/<name>/main.go`, router (`chi`/`gin`/`echo`/`fiber`/`mux`/`net/http`), handler dir, repo/store dir, sql driver (`pgx`/`sqlx`/`sqlc`/`gorm`) | (a) `flowchart LR`: `main` → router → middleware (`chi.Use(...)`) → handler → service → repo → DB (label engine). (b) `sequenceDiagram` for one route covering context propagation, query, response. (c) If goroutines/channels orchestrate work, add `flowchart LR` of goroutine → channel → consumer. (d) `classDiagram` of sqlc/`ent` generated models if present. |
| **Go CLI (cobra)** | `cobra.Command{}` tree, `cmd/root.go`, subcommand files | `flowchart TD` of command tree → flag parsing → `RunE` body → side effects. Each subcommand a node. |
| **Rust web service (axum/actix/rocket)** | `Cargo.toml` w/ framework dep, `#[tokio::main]`, `Router::new().route(...)` or `#[get(...)]` macros, `sqlx`/`diesel`/`sea-orm` | (a) `flowchart LR`: `main.rs` → router → tower layers / extractors → handlers → service → repo → DB (engine-labeled). (b) `sequenceDiagram` for one route showing extractor parse → handler `async fn` → `await` query → response. (c) `classDiagram` of derived `Queryable`/`FromRow` structs. |
| **Rust CLI (clap)** | `#[derive(Parser)]`, `enum Commands`, subcommand match arms | `flowchart TD` of `Commands` enum → match arm → function → side effects. |
| **Rust WASM frontend (leptos/yew/dioxus)** | `trunk.toml`, `wasm-bindgen`, component macros (`#[component]`) | `flowchart TD` of component tree + signal/state flow. |
| **htmx-driven hypermedia app** | `hx-*` attributes in templates, server returns HTML fragments, no SPA framework | (a) `flowchart LR`: template attribute (`hx-get="/foo"`) → server route → handler → template partial → swap target in DOM. (b) `sequenceDiagram` of one interaction: user click → htmx request → server handler → partial HTML → `hx-swap` into target. Pair with the backend archetype (Go/Rails/Django/Laravel/Flask). |
| **Alpine.js layer** | `x-data` islands, `Alpine.store`, sprinkled on a server-rendered or static site | `flowchart TD` of each Alpine component: `x-data` state → `x-on`/`@click` handlers → reactive bindings → DOM effects. List `Alpine.store` globals as a separate subgraph. Always emitted alongside the host archetype (WordPress / Laravel Blade / Rails ERB / static HTML). |
| **Laravel + Livewire** | `livewire/livewire` composer dep, `app/Livewire/*.php` extending `Livewire\Component`, Blade views with `wire:*` attrs | (a) `flowchart LR`: Livewire component class → public properties / `render()` → Blade view → `wire:` directive → HTTP roundtrip (`livewire/update`) → component method → DB → re-render. (b) `sequenceDiagram` of one `wire:click` showing hydration → method invocation → component diff → DOM patch. (c) `classDiagram` of component classes with public props, computed properties, listeners, and methods. Pair with the broader Laravel archetype. |
| **Vue.js SPA / SFC tree** | `.vue` files with `<script setup>`, `vue-router`, Pinia stores, fetch composables (`useFetch`, `useFetchClient`) | (a) `flowchart TD` route tree from `createRouter`. (b) `flowchart LR` for each page: page SFC → child components → composable → fetcher → backend endpoint. (c) `classDiagram` (or `flowchart LR`) of Pinia stores: state → getters → actions → mutations → consumers. |
| **Spring Boot / Java app** | `@SpringBootApplication`, controllers/services/repos, JPA entities, `application.yml` | (a) `flowchart LR`: `main` → DispatcherServlet → filter chain → `@RestController` → `@Service` → `JpaRepository` → DB. (b) `sequenceDiagram` for one endpoint covering validation (`@Valid`), transaction (`@Transactional`), JPA query, response mapper. (c) `classDiagram` of `@Entity` classes with relations (`@OneToMany`, `@ManyToOne`). (d) If `@KafkaListener` / `@RabbitListener` present, `sequenceDiagram` of producer → broker → listener → DB. |
| **Other Java HTTP (Quarkus/Micronaut/JAX-RS/Vert.x)** | Framework-specific annotations from Phase 1 | Same shape as Spring Boot using the framework's actual classes (e.g. `RoutingContext` for Vert.x). |
| **ASP.NET Core (Minimal API or MVC)** | `Program.cs` with `WebApplication.CreateBuilder`, controllers/Minimal API mappings, `DbContext` | (a) `flowchart LR`: `Program.cs` → middleware pipeline (`app.Use…`) → endpoint routing → handler/controller → service → EF Core `DbContext` → DB. (b) `sequenceDiagram` for one endpoint covering model binding, validation, EF query, response. (c) `classDiagram` of EF entities with navigation properties. (d) DI registration map: `flowchart LR` of `Services.AddScoped/Singleton/Transient` → consumers. |
| **Blazor app (.NET)** | `*.razor`, `App.razor`, `_Imports.razor`, `@code {`, SignalR for Server, WASM project for Client | (a) `flowchart TD` component tree by routable `*.razor` page. (b) For Blazor Server: `sequenceDiagram` of UI event → SignalR → server component → render diff → DOM patch. For WASM: same as SPA. |
| **C# desktop (WPF / WinForms / MAUI)** | `*.xaml` files, code-behind `.xaml.cs`, MVVM ViewModels, `INotifyPropertyChanged` | `classDiagram` of View → ViewModel → Model bindings; `flowchart LR` of command pipelines. |
| **Unity game (C#)** | `MonoBehaviour` subclasses, `Assets/Scripts/`, scene `.unity` files | `flowchart LR` of `Update()` loop per script + cross-script messaging (`SendMessage`, events, ScriptableObjects). |
| **C++ service / app** | `CMakeLists.txt`, framework (Qt/Drogon/Crow/Boost.Beast/oat++) or plain `main.cpp` | (a) `flowchart LR` build target graph from `add_executable`/`add_library`/`target_link_libraries`. (b) `flowchart TD` per executable: `main` → init → event loop / dispatch → modules → IO. (c) `classDiagram` for the core domain classes (capped at 25 by inbound edges). (d) For Qt apps: `flowchart LR` of `signals`/`slots` wiring per `QObject`. |
| **C++ library / header-only** | `include/<lib>/*.hpp`, no `main`, `add_library` in CMake | `classDiagram` of public API classes, plus `flowchart LR` of internal include graph. |
| **C application / system module** | `main.c` + `Makefile`, or kernel/embedded SDK markers | (a) `flowchart TD`: `main` → arg parsing → init → main loop → cleanup. Highlight `goto cleanup;` error paths. (b) `flowchart LR` of include graph for top 20 `.c` files by inbound includes. (c) For kernel/embedded: `flowchart LR` of init macros → IRQ/handler dispatch → device tree / peripheral I/O. |
| **TypeScript-typed Node backend** | Same backend signals as Node above, plus `tsconfig.json`, Zod/tRPC/Nest/typed Prisma | Same diagrams as the Node archetype, with one addition: `classDiagram` of inferred / declared types at API boundaries (request DTOs, response DTOs, ORM model types) and how they flow across layers. If tRPC, render the procedure tree as `flowchart TD` (router → procedure → input schema → resolver → output type). |
| **TypeScript-typed frontend** | `.tsx` / `.vue` (lang=ts) / `.svelte` (lang=ts) | Same as the corresponding framework archetype, with an added `classDiagram` of prop/state interfaces and a `flowchart LR` showing how generated types (Prisma, GraphQL Codegen, OpenAPI) flow from schema → generated module → consumers. |
